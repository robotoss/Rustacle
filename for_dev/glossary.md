# Glossary

> Single source of truth for every load-bearing term in Rustacle. If a term appears in another doc and isn't defined here, fix it here first.

## Core concepts

**Kernel** — the `rustacle-kernel` crate. Owns plugin lifecycle, IPC routing, event bus, permission broker. Knows nothing about specific plugins.

**Plugin** (a.k.a. **Module**) — a self-contained unit of functionality. Default form: a signed WebAssembly Component (`.wasm`) loaded by `rustacle-wasm-host`. Exceptional form: a whitelisted native crate (currently only `plugins/terminal`).

**Host** — the Rust process (`rustacle-app`) that embeds the kernel and the WASM runtime. "Host functions" are Rust functions exposed to WASM plugins.

**Manifest** — `ModuleManifest`: declarative metadata shipped by each plugin, listing version, required capabilities, UI contributions, event-bus subscriptions. Read during `negotiate_capabilities`.

**Capability** — a typed permission a plugin asks the user for. Example variants: `Fs { scope, mode }`, `Net { allow_hosts }`, `Pty`, `Secret { key }`, `LlmProvider`. Granted or denied via the Settings UI; enforced by the Permission Broker.

**Permission Broker** — the single chokepoint on every capability use. Plugins never self-authorize; they call a host function and either proceed or receive `PermissionDenied`.

**Event Bus** — topic-typed pub/sub inside the kernel. Publishers declare `Topic<T>`; subscribers receive via bounded `mpsc` channels with per-topic backpressure policy (`BlockPublisher | CoalesceLatest | DropOldest`).

**IPC** — the typed command/event surface between the UI (TS) and the kernel (Rust), generated by `specta + tauri-specta` into `bindings.ts`.

## Agent concepts

**Turn** — one user-message → agent-answer cycle. Owns a `TurnId`, a `CancellationToken`, and a sequence of `ReasoningStep`s.

**Step** — one `ReasoningStep` inside a turn. Kinds: `Thought`, `ToolCall`, `ToolResult`, `PermissionAsk`, `Answer`, `Error`. Each has a `StepId` and optional `parent_id` (for sub-agent trees).

**Thinking loop** (a.k.a. **Harness loop**) — the single async loop that runs one turn: prompt assemble → LLM stream → tool dispatch → observation → repeat. Defined in `agent_reasoning.md` §1 and §5.

**Tool** — a typed capability exposed to the LLM. Has `name`, `description`, `input_schema` (JSON Schema), `validate`, `check_permissions`, `prompt()` (tool-level system prompt contribution), and `call()`. Tool catalog lives in `tools_catalog.md`.

**Tool dispatch table** — the registry `(tool_name → handler)` built at plugin init. The harness loop never string-matches tool names outside this table.

**Concurrent tool** — a tool marked safe to run in parallel with other concurrent tools inside the same turn (e.g., `FsRead`, `Grep`). Inspired by cc-src's `StreamingToolExecutor` pattern.

**Serialized tool** — a tool that must run alone (e.g., `Bash`, `FsEdit`, `FsWrite`). The harness drains all in-flight concurrent tools before executing one.

**Model profile** — a user-configured bundle of `(provider, model_id, temperature, max_tokens, system_prompt_overlay, enabled_tools)`. Selected in the Settings UI; no file editing.

**Provider** — an implementation of the `LlmProvider` host trait (OpenAI, Anthropic, Ollama, LM Studio, llama.cpp-server, …). Providers live in host crates; the agent plugin calls them through a host function, never directly.

**Prompt assembly** — the deterministic 8-layer function `assemble_prompt(ctx) → Prompt`. See `agent_reasoning.md` §3.2. Must be byte-identical for identical inputs.

**System base** — the immutable top layer of every prompt. Full text lives in `prompts_catalog.md`.

**Env context** — the layer derived from the active terminal tab (`cwd`, shell, OS, open tabs summary). Regenerated per turn.

**Project docs** — `RUSTACLE.md` / `CLAUDE.md` files walked up from `cwd`. Injected as system messages.

**Memory** — long-term user memory, SQLite-backed, scored against the current user turn, top-K injected.

**Cancel token** — `tokio_util::sync::CancellationToken` owned by a turn. The UI Stop button flips it; every awaited future is cancel-safe.

## Terminal concepts

**Tab** — one shell session with its own PTY, `cwd`, shell, env, command history buffer, and agent sub-context.

**Tab group** — a set of tabs sharing a project context (usually a common root).

**Split** — recursive horizontal/vertical division within a tab. Each leaf has its own PTY.

**Tool-use redirection** — the UI affordance by which the user reassigns a pending `ToolCall` from the active tab to another tab by dragging the tool-call card.

**Directory-aware title** — a tab title computed from the nearest project root (git toplevel, `Cargo.toml`, `package.json`, `RUSTACLE.md`).

## Storage concepts

**AppState** — the kernel-owned `struct` accessed via Tauri's `State<AppState>`. Holds plugin registry, event bus handles, permission broker, settings store.

**Settings store** — typed, SQLite-backed key-value map fronting the Settings UI. Import/export round-trips through this schema.

**BlobRef** — opaque handle to a blob stored under the app data dir. Used for large reasoning payloads to keep SQLite small.

**Trace ID** — `ulid` string stamped on every IPC command and tracing span. Shown in user-facing error dialogs so support can correlate.

## Process / lifecycle

**Discover** — the kernel scans the plugins dir and loads each `.wasm` (or whitelisted native crate) metadata.

**Verify signature** — each plugin ships with an Ed25519 signature; the host verifies against a trusted key list before instantiation.

**Negotiate capabilities** — the kernel presents the plugin's requested capabilities to the user (via Settings UI) and records grants.

**Init** — the plugin's `init(ctx)` runs, wiring state and subscribing to topics.

**Hot-swap** — replacing a running plugin with a new version: spawn new instance → drain in-flight commands on old → atomic registry swap → drop old. State migration policy per-plugin (see `architecture.md` §4.4).

**Suspend** — a plugin can be paused: subscriptions buffered, commands rejected with `Unavailable`, resources retained.

## Security

**Sandbox** — the wasmtime Store with fuel metering, memory limits, and a narrow linked host interface. One plugin instance = one Store.

**Fuel metering** — wasmtime's per-instruction budget that traps runaway guests.

**Host function** — a Rust function linked into a WASM plugin's import table. The set is small, audited, and documented in `project_structure.md`.

**Secret** — a value stored in the OS keyring via the `keyring` crate, wrapped in `SecretString`, redacted on `Debug`, excluded from settings export.
